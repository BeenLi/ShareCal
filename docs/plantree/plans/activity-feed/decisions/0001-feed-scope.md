# Decision 0001 — 动态流内容范围 = 仅评论互动

- **日期**：2026-06-14
- **状态**：Accepted（用户确认）

## 决定
「动态」tab 只承载**评论互动**（对方对我日程的评论 + 我的回复）。**不**并入邀请、历史访问请求。**邀请 tab 保留为独立入口**。

## 理由
- 用户原话把动态放在「日历」和「邀请」之间，意味着邀请 tab 继续存在。
- 聚焦单一互动语义，最快落地、与现有邀请 tab 职责不重叠。

## 影响
- 数据源仅 `EventComment`（已 CloudKit 同步、含 `isRead`）。
- 邀请角标逻辑（`PendingActionBadgePlan`）保持不变；动态未读需独立计算（见 plan OQ4）。

## 备选（未采纳）
- 「评论 + 邀请」合并流 / 「统一动态中心」（评论+邀请+访问请求）：信息更全但与邀请 tab 职责重叠、复杂度更高。
